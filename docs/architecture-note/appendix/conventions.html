規約とテスト方針

コードベースの一貫性を保つための約束と、2 系統のテストの回し方。

コーディング規約（抜粋）

  • 状態管理: @Observable のみ。ObservableObject/Combine は使わない。
  • 並行性: アプリは既定 @MainActor。バックグラウンドのコールバックは Task { @MainActor in } で戻す。
  • 設定キーの一元化: @AppStorage/UserDefaults のキーは各パッケージの専用 enum に集約（GridSettingsKeys / CacheSettingsKeys / DropboxCacheSettingsKeys / BackupSettingsKeys / AutoAlbumSettingsKeys / アプリ層は AppSettingsKeys）。
  • ロギングの一元化: MosaicSupport.LogChannel に集約。error は常時、info/verbose は DEBUG のみ。各レベルは診断ログにも転記。
  • DI seam: 外部依存はプロトコル化しデフォルト引数で本番実装を注入（HTTPClient / DateProvider / AccessTokenProvider）。
  • UI 文言は英語: 表示文字列は英語でハードコードしない方針（国際リリース前提）。コメント・コミットは日本語可。
  • 文字コード: ソース・API ボディ・Keychain 値はすべて UTF-8。
  • 機密の非保存: クレデンシャル・トークン・APIキーをファイルやログに書かない。

テストの 2 系統

系統対象実行
FAST（macOS）Foundation のみの純ロジック（PhotoSourceKit / AutoAlbumCore / BackupKit など）swift test
iOS SimUIKit / SwiftData / Photos 依存（DropboxCore / PhotosFeatureKit）xcodebuild test -destination 'platform=iOS Simulator,...'
# 全テスト一括（fast = macOS のみ / ios = シミュレータ / all = 両方）
scripts/test.sh all

# 個別（純ロジック）
cd Packages/AutoAlbumCore && swift test

# アプリのビルド
xcodebuild -project MosaicPhotos.xcodeproj -scheme MosaicPhotos -sdk iphonesimulator build
落とし穴 ローカルパッケージに新規ファイルを足すと、依存先の .build が stale になり「型が見つからない」で swift test が落ちることがある。rm -rf Packages/*/.build で解消。

検証ポリシー

  • シミュレータのアプリ画面スクショは撮らない（個人写真が写り得る）。検証はビルド・ログ・ユニットテスト・コンテナ内ファイル件数で行う。
  • 破壊的操作（消去・初期化・上書き）はユーザー環境に対して事前確認。検証は使い捨て環境で。
  • git push は明示指示があるときだけ。作業は基本 main 上で行いブランチは作らない。
