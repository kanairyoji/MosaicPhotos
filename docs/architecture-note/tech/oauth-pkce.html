OAuth2 + PKCE

他サービス（Dropbox）の認可を、パスワードを預からずトークンで得る仕組み。PKCE はネイティブアプリのための横取り対策。

OAuth 2.0 認可コードフロー

  1. アプリがブラウザでサービスの認可画面を開く。
  2. ユーザーが許可すると、認可コードがリダイレクトで返る。
  3. アプリがコードをアクセストークンと交換する。
  4. 以降はトークンで API を呼ぶ。期限切れは refresh トークンで更新。

なぜ PKCE が要るか

ネイティブアプリはクライアントシークレットを安全に隠せない。カスタム URL スキームのリダイレクトは他アプリに横取りされ得る。PKCE（Proof Key for Code Exchange）は使い捨ての秘密でこれを防ぐ。

sequenceDiagram participant App participant Auth as 認可サーバ App->>App: code_verifier をランダム生成 App->>App: code_challenge = SHA256(code_verifier) App->>Auth: 認可要求（code_challenge を添付） Auth-->>App: 認可コード App->>Auth: コード + code_verifier を送信 Auth->>Auth: SHA256(verifier) が challenge と一致するか検証 Auth-->>App: アクセストークン

コードを横取りしても code_verifier を知らなければトークンに換えられない。

このアプリでの実装

  • ASWebAuthenticationSession で認可画面を表示し、MosaicPhotos://oauth/dropbox でコールバックを受ける。
  • PKCEGenerator が verifier/challenge を生成（純ロジック・テスト対象）。
  • トークンは Security（Keychain）に保存。appKey/redirectURI はアプリ層の設定に置き、Core は値を持たない。
  • 機密値はログ・ファイルに書かない（セキュリティ方針）。
