診断と堅牢性

Mac/Console が使えない実機でも原因を追えるよう、端末上に診断ログを残す。SwiftData の起動時クラッシュは自己修復で回避する。

端末診断（Diagnostics）

アプリの init() で Diagnostics.install() を呼ぶと、次を Caches/diagnostics.log に記録する。

  • 未捕捉 ObjC 例外（NSSetUncaughtExceptionHandler）— 名前・理由・コールスタック。
  • メモリ圧迫（DispatchSource.makeMemoryPressureSource）— warning/critical とフットプリント。
  • LogChannel の各レベル — error は Release でも、info/verbose は DEBUG のみ転記。

Developer Options の DiagnosticsLogView で閲覧・共有・クリアできる。

注意 Swift の fatalError や SwiftData の trap はこのハンドラを通らず、標準クラッシュログ側に出る。

自己修復する ModelContainer

SwiftData の ModelContainer 初期化は、ストア破損やスキーマ不整合のとき起動時に trapし、実機で原因不明のクラッシュになりやすい。そこで makeResilientContainer(...) が段階的にフォールバックする。

.store / -wal / -shm] C --> D[再生成] D --> E{成功?} E -->|はい| OK E -->|いいえ| F[インメモリで起動] F --> OK

データは失うが起動は止めない（失敗は診断ログへ）。AutoAlbumStore / DropboxCacheStore / BackupEngine が採用。

メモリ圧迫への対応

MosaicSupport.MemoryPressureMonitor が圧迫フラグを保持し、Diagnostics のメモリ圧迫ソースが warning/critical で立てる（一定時間後に自動解除）。背景の CLIP 埋め込みはこのフラグを見て一時停止し、jetsam を避ける。

CLIP の可視化

MobileCLIP.modelsBundled でモデル同梱の有無をロードせず判定でき、Developer Options に表示する。ランタイムのロード結果（成功/失敗/未同梱・所要時間・フットプリント）も診断ログへ残す。
