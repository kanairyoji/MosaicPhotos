キャッシュ戦略

画像キャッシュのプリミティブ（メモリ + ディスク I/O）を ImageCacheKit に共通化し、端末写真と Dropbox の双方が利用する。破棄ポリシー（LRU）は利用側が持つ。

2 層キャッシュ

graph TD REQ[サムネ/フル画像要求] --> MEM{MemoryImageCache
NSCache} MEM -->|hit| RET[即返す] MEM -->|miss| DISK{DiskImageStore} DISK -->|hit| LOAD[読み込み→メモリへ→返す] DISK -->|miss| FETCH[ソースから取得→両層へ保存]
  • MemoryImageCache — NSCache ラッパー（メモリ層）。
  • DiskImageStore — ディレクトリ単位のディスク I/O と LRU 列挙。コアは Foundation のみ（macOS テスト可）、UIImage 便宜は #if canImport(UIKit)。

利用側ごとの破棄ポリシー

利用側メタ管理破棄
LocalPhotoCore の ThumbnailCache（actor）—LRU（独自）
DropboxCore の DropboxCacheStore（actor）SwiftData（CachedDropboxItem / CacheUsageEntry）CacheUsageEntry（最終アクセス）ベースの LRU・容量上限

Dropbox キャッシュの無効化

ファイルの contentHash 変更を検知してエントリを無効化する。バイナリは Caches/DropboxKit/{thumbnails,fullimages}/ にハッシュ化ファイル名（DropboxCacheNaming）で保存。

上限はユーザー設定

サムネ/フル画像の上限は設定（Storage 画面）から変更でき、キーは各パッケージの専用 enum（CacheSettingsKeys / DropboxCacheSettingsKeys）に集約する。

設計判断 メモリ層と I/O プリミティブを 1 つのパッケージに集約しつつ、破棄ポリシーは各利用側が持つ。端末写真と Dropbox では無効化条件（contentHash の有無）も容量管理（SwiftData の有無）も異なるため、共通化しすぎず「プリミティブ共通・ポリシー個別」に留めている。
