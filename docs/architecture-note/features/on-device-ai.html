オンデバイス AI

固定の語彙リストや OCR を持たず、任意言語の自然文で AI アルバム／意味検索を行う。実装は「タグ台帳＋LLM 審査」の多層構成（ADR-23/24）：シーンタグ・CLIP 埋め込み・VLM キャプションを夜間に付与し、検索は決定的ハード条件＋タグ一致＋CLIP 対比＋字句の RRF 融合＋証拠ゲート＋LLM 審査で決める。通信なし・API キー不要。モデルと検索品質はすべて実測で決める（records/model-evaluations.md）。

索引（夜間パイプライン）と検索

索引は夜間バッチ（電源＋Wi-Fi＋アプリ非使用の BGProcessingTask・実行タイミング参照）で、新しい写真から先に（撮影日降順・ADR-36）次の順に付与する。

  1. Vision シーンタグ（OS 内蔵 VNClassifyImageRequest・約 1,300 クラス・hasMinimumRecall(forPrecision:) の校正済み足切り＝自前閾値が消える）— 全写真
  2. CLIP 埋め込み（OpenCLIP ViT-B-32・INT8 量子化・意味検索の索引）— 全写真
  3. VLM キャプション（SmolVLM-500M・英語一文・LLM 審査の「目」）— お気に入りを中心とした重要写真（最重量パスの集中投下・ADR-34。AI アルバム審査時は候補上位にもオンデマンド生成し、生成分は台帳へ永続化＝ADR-35）

なぜ「閾値レス」へ再設計したか（ADR-24） 以前は生 CLIP コサイン＋絶対閾値だったが、ユーザーのライブラリ分布が事前に不明で閾値を調整できず「除外が 97% を落とす」「意味採用 0 件」が起きた。そこでタグ一致（離散・閾値レス）を第 3 のランキングに加え、除外は肯定/否定ベクトルの相対判定（絶対閾値なし）にし、最終判断をLLM 審査（言語判定）に委ねた。判定根拠がタグ/キャプションで可視化されデバッグ可能になる。
解釈は「作成時に 1 回だけ」＋言い換えプローブ（ADR-23・ADR-35） 検索文→QuerySpec・英訳・言い換えプローブ（最大 4）の LLM 解釈はアルバム作成/編集時に 1 回だけ実行して AIAlbumInterpretationStore（JSONFileStore・版管理）へ永続化する。起動時・写真追加時に LLM は走らない。意味採点は主フレーズ＋プローブの max-over-probes＝言い換え表現の取りこぼしを回収（ハーネス実測で言い換え再現率 +17〜18pt）。ただし除外つきアルバムではプローブを対比に使わない（肯定の最大値が底上げされ除外が素通りする回帰の防止）。再評価は増分（新規埋め込み分だけ採点）＋ドリフト検知のフル再評価。

量子化の非対称性（学び） 出力が連続ベクトルの部品（CLIP・VLM 視覚エンコーダ）は INT8 に強い（fp16 と cos≈0.999）。一方 LLM デコーダは弱い（次単語 argmax が反転・fp16 と一致 26%）。よって CLIP は全体 INT8、SmolVLM は視覚のみ INT8・デコーダ fp16 とした。また Core ML の正しさは Mac だけでは検証できない——Florence は Mac の全 compute unit で正常でも実機 fp16 で破綻した。新モデルは実機ログ（生成 ID・有限性）で確認してから採用する。

検索の融合（RRF）と証拠ゲート・審査

検索は 4 層。決定的なハード条件で絞り、複数のランキングを Reciprocal Rank Fusion で束ね、除外つきは証拠を要求し、最後に LLM が言語で判定する。

  • P0 接地: 日付は RelativeDateParser（日英）を唯一の出典・場所はカタログ/原文出現に接地・人物名は顔クラスタへ接地（PersonNameGrounder＝「太郎」→山田太郎・姓名の部分指定対応）。人物条件の評価は焼き込みでなく現在のクラスタ名でライブ照合（改名/統合が即反映）。QuerySpecSanitizer がプレースホルダ除去・include/exclude 衝突解消。
  • P1 融合: Vision シーンタグ一致（離散・閾値レス）＋CLIP 対比（max-over-probes・除外は相対判定のみ）＋LexicalSearch（地名/人物）を AIAlbumSearcher が RRF 融合。除外語はタグの離散一致でもハード除外。
  • P2 証拠ゲート＋審査: 除外つきはタグ/顔実測/キャプションの証拠を必須にし、キャプションの無い候補上位にはその場で生成（予算 40 枚/審査・ADR-35）してから AlbumVerifier（Foundation Models）が keep/drop/unsure 判定。unsure は最大 2 回の再判定で多数決。空振り時はプローブ語を生成して 1 回だけ再検索（Refine）。
検索品質は回帰ハーネスで数値管理 SearchQualityTests（Imagenette 200 枚＋正解ラベル付き 28 クエリ・実パイプライン実行）で Recall@20 / 精度 / 再現率を計測する。ベースライン（精度 0.99・再現率 0.76）で「言い換えの取りこぼし」が弱点と数値で特定し、マルチプローブ導入の A/B（再現率 0.76→0.86）も同ハーネスで検証した。実行手順は model-evaluations.md §4。

作成 UI の入力支援（ADR-37）

コンポーザーは本番検索と同じ決定的レイヤーで入力を支援する（＝プレビューが実際とズレない）。(1) サジェストチップ＝ライブラリから観測された語（命名済み人物・カタログ実在の地名・頻出タグ∩レキシコンの日本語・日付定型）をタップで挿入——確実にヒットする語だけを出す。(2) 接地プレビュー＝入力がどう解釈されたか（人物/場所/視覚語/日付）を色付きチップで表示。(3) ハード条件のヒット件数をライブ表示（0 件は作る前に分かる）。基盤はライブラリスナップショット（5 分キャッシュ）＋350ms debounce。

AutoAlbumCore はプロトコルだけを定義し、Core ML / Vision / Foundation Models 依存の実体は MobileCLIPKit が実装する。アプリの AutoAlbumAdapters が結線する。

ピープル（顔クラスタ）

写真アプリの「ピープル」は公開 PhotoKit API でアクセスできない（旧 subtype-1000 方式は常に空だったため撤去）。代わりに Vision 顔検出（VNDetectFaceRectanglesRequest）＋同梱顔モデル（facenet・512 次元 L2 正規化）で identity 埋め込み→逐次クラスタリングで自前の「人物」を作る。ロジックは AutoAlbumCore/Faces（FaceClustering 純・FaceStore は別コンテナ "FacesV1"・PeopleEngine ファサード）。対象は端末＋クラウド（Dropbox）両方（ADR-33）：端末は 640px、クラウドはキャッシュ済みサムネイル（追加DL なし）で検出——低解像度ぶん大きく写った顔中心という割り切り。長押しメニューから改名・代表写真・顔の付け替え・同一人物の統合（ADR-29）。命名した人物は AI アルバムの人物条件に接地し、照合はライブ（改名/統合が即反映）。顔モデル未同梱ならセクション非表示。

表示タグ・説明文は検索と同じ台帳

フル画像に出すキーワードタグは Vision シーンタグ（TagStore・検索の一次ランキングと同一ソース）を第一に、CLIPDisplayLabeler（約 300 語ゼロショット・保存済み埋め込み使用）で補完する。VLM キャプションは「AI description」欄に表示（お気に入り＝生成対象で未生成なら「充電中に生成されます…」を表示）。顔の数・スクリーンショット判定・解析状態も同パネルに出す。進捗はユーザー向けの「AI 解析の状況」（パス別進捗・最終実行時刻・今すぐ解析・説明文の確認画面）で可視化する。

背景処理のスロットリング

シーンタグ・CLIP 埋め込み・顔スキャンは BackgroundTrickle の骨格で小バッチ＋バッチ間スリープ＋.background QoS で trickle 実行する。CLIP 埋め込みと VLM キャプションはインターリーブ（ADR-30・逐次だとキャプションが永遠に未着手になる実障害の修正）。重さは設定（段階）で調整でき、停止判定は 1 枚単位（ADR-25）。ユーザー操作中・メモリ圧迫中・クラウド取得中は処理を譲る。シミュレータでは背景推論を実行しない。詳細は バックグラウンドと省電力。
