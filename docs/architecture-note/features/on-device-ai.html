オンデバイス AI

固定の語彙リストや OCR を持たず、CLIP のオープン語彙で自然文検索を行う。通信なし・APIキー不要。クエリはまず英語へ翻訳してから埋め込む。

採用モデルと認識率ベンチマーク

埋め込みには OpenCLIP ViT-B-32（DataComp 学習）を Core ML 化して使う。複数候補を、出荷する Core ML モデルそのままで認識率を測って選定した（scripts/eval_recognition.sh）。

評価条件と選定 ImageNet-1k 1000クラスのゼロショット分類（各クラス20枚=200枚）と自然文クエリ10件を、CoreMLTools で CPU 実行して測定。採用は ViT-B-32/datacomp＝軽量（patch32・画像エンコーダ ~60MB）ながら 75% を達成し、自然文クエリは満点。ViT-B-16 は同等精度だが画像推論が約4倍重く、67k 枚の背景埋め込み（電池/時間）と相性が悪いため見送り。モデル変更時は perceptionVersion を採番して全再埋め込みする。

なぜ「語彙ゼロ」なのか

設計判断 固定タグ方式（あらかじめ決めた数百語で分類）は語彙外の検索に弱い。CLIP は画像とテキストを同じベクトル空間に埋め込むため、「赤い傘」「夕焼けの海」のような自由な文を、語彙リストに無くてもコサイン類似で照合できる。これがオープン語彙検索。OCR・固定タグは持たない。

翻訳を前段に置く理由

CLIP（OpenCLIP ViT-B-32）のテキストエンコーダは英語中心。日本語クエリをそのまま入れると精度が落ちる。そこで Foundation Models（オンデバイス LLM）でクエリを英語へ正規化してから CLIP に渡す。翻訳もオンデバイスなので通信不要。

検索の融合（RRF）

意味検索（CLIP）だけでなく、構造化条件（日付・場所・人物）と字句一致（地名・人物名）も併用し、Reciprocal Rank Fusion で順位を統合する。これにより「去年 京都で 撮った 紅葉」のような複合クエリに強くなる。

  • QueryUnderstanding — クエリを構造化条件＋意味語に分解（ルールベース / Foundation Models）。
  • SemanticRanker / ClipMath — コサイン類似で意味的に並べ替え。
  • LexicalSearch — 地名・人物の字句一致。
  • AIAlbumSearcher — 上記を RRF で融合し最終結果に。

知覚 seam（プロトコルと実装の分離）

AutoAlbumCore はプロトコルだけを定義し、Core ML 依存の実体は MobileCLIPKit が実装する。アプリが結線する。

プロトコル（AutoAlbumCore）実装（MobileCLIPKit）

表示タグは検索と分離

フル画像に出すキーワードタグは CLIPDisplayLabeler が約 300 語の英語キーワードに対するゼロショット（保存済み埋め込みを使う）で生成する。これは表示専用で、検索は語彙ゼロのまま縛らない。

背景埋め込みのスロットリング

全写真の画像埋め込みは PhotoTagger.embedUnprocessed が小バッチ＋バッチ間スリープ＋.background QoS で trickle 実行する。重さは設定（段階）で調整でき、ユーザー操作中・メモリ圧迫中は処理を譲る。
