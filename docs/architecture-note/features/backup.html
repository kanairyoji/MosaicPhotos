バックアップ

端末写真を Dropbox 上の指定フォルダへアップロードする。差分判定・人物/アルバム情報の付帯・上限制御を BackupKit が担う。

構成

型役割
BackupEngine@MainActor @Observable。バックアップ全体のオーケストレーションと進捗（phase）。
DropboxBackupUploader写真/メタデータの HTTP アップロード。認証・SwiftData から独立しテスト対象。
BackupAssetReaderPHAsset 本体データの取得。
BackupIndexingPeople / Album インデックス構築（top-level・Task.detached 用）。
BackupPlanningアップロード差分算出・エラー要約の純ロジック（テスト対象）。

差分とメタデータ v2（ADR-38）

アップロード済み ID は BackupAssetRecord（SwiftData）で管理し、再実行時は未送信分だけを送る。1 回あたりの上限（10/50/100/500/無制限）を設定でき、BackupPlanning が純関数として差分を決める。

メタデータは v2 ＝カタログ＋撮影月シャード（ADR-38）。将来のオフロード（バックアップ済みローカル写真の削除）を見据え、端末を削除すると再生成できない情報を写真ごとに保全する：アルバム所属・お気に入り・撮影日（v1 から）に加え、人物名（顔クラスタのユーザー命名）・GPS（EXIF に無い写真の唯一の出典）・localIdentifier（ローカル⇔クラウド対応表）・スクリーンショット判定・VLM キャプション（v2 で追加・すべて Optional＝v1 と相互に読める）。

  • .mosaic/catalog.json — 入口（スキーマ版・シャード一覧・アルバム/人物カタログ・数 KB）。
  • .mosaic/meta/<YYYY-MM>.json — 撮影月シャード。バックアップ 1 回で更新するのは触った月だけ（download→merge→upload）。旧 v1 の「全エントリを毎回書き直す」（6 万枚で 15〜25MB/回）を有界化した。
  • v1 metadata.json は凍結。読み込み（DropboxPhotoStore.loadBackupMetadata）は「v1 ベース → v2 シャードで上書きマージ」で統合する。
  • 人物名・キャプションは BackupKit から AutoAlbumCore へ依存せず、seam（peopleNamesProvider / captionsProvider）で Composition Root（AutoAlbumAdapters）が結線する。

オフロード台帳と端末アルバムの合成表示（ADR-39）

将来のオフロード（バックアップ済みローカル写真の検証つき削除）に備え、台帳と表示合成を先行実装している。OffloadRecord（SwiftData）が「アプリが削除した写真」の localIdentifier / Dropbox パス / 所属アルバムを記録し、端末アルバムを開く DeviceAlbumPhotosView が「PHAssetCollection 現存メンバー＋台帳のクラウド代替」をメンバー限定 MergedPhotoStore で混在表示する（撮影日ソートで元の位置に混ざる）。補完の条件を「台帳にある」に限定するのが要点＝ユーザーが写真アプリで意図的に削除した写真を蘇らせない。再インストール時は metadata v2 の offloadedAt マーカーから台帳を再構築する。台帳が空（現状）のときの表示は従来と完全に同一。

端末フォルダ分離（家族共有・ADR-41）

1 つの Dropbox アカウントを家族で共有する想定で、バックアップは <ルート>/<表示名>-<短ID>/（例 /MosaicPhotos/iPhone-3F2A8C/）に保存する。ファイルだけでなく .mosaic メタデータも端末ごとに分離され、download→merge→upload 方式のメタデータが端末間で後勝ち競合する問題（ADR-38 の単一端末制約）が構造的に消える。短 ID は初回生成して Keychain に永続化（再インストール後も同一フォルダ）。既存のフラットな旧ファイルは移動せず、メタデータ読み込みは「ルート＋端末フォルダ」の複数ルート統合で互換を保つ。

検証つきアップロードとオフロード（ADR-40）

アップロードの「済み」判定は content_hash の一致を必須にした：ローカルで Dropbox の content_hash（4MB ブロック SHA-256 連結 → SHA-256・DropboxContentHash 純関数）を計算し、files/upload 応答の hash と一致して初めて記録する（HTTP 200 でも不一致なら済みにしない）。409（同パス既存）は get_metadata で同一性を確認し、不一致（同名の別写真）は autorename で別名アップロードする——旧実装は無確認で「済み」にしており、オフロードでの永久喪失リスクだった。

オフロード（検証つき削除）は「削除は証明の後」を不変条件に：直前再検証（hash/サイズのリモート実測一致・編集検知・Live Photo 除外）→ 台帳記録 → PhotoKit 削除（OS 確認ダイアログ必須・「最近削除した項目」30 日）→ キャンセルならロールバック → metadata マーカー。UI はドライラン既定（何も消さない検証一覧）で、実削除は Developer Options のゲート＋1 回の上限つき。安全性は OffloadSafetyTests（偽 Dropbox＋削除モック）が「消してはいけない全ケースで削除要求ゼロ」を機械的に保証する。

純ロジックの切り出し

設計判断 差分算出（BackupPlanning）とアップロード（DropboxBackupUploader）を、認証や SwiftData から切り離した純ロジック/単機能にしている。これにより macOS の swift test で「どの写真を送るか」「エラー要約」をネットワークなしで検証できる。

UI と診断の分離

通常設定は BackupSettingsView、詳細診断（進捗・フォルダ確認・統計・ログ）は BackupDebugSection に分け、後者は Developer Options 画面が合成して表示する。
