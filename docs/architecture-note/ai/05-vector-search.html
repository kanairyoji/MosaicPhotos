5. ベクトル検索と近傍探索

クエリの埋め込みに「近い」ベクトルを大量の候補から見つけるのがベクトル検索。小規模なら全件計算、大規模なら近似（ANN）を使う。

距離・類似度

指標式の直感用途
コサイン類似向きの一致（-1〜1）埋め込み比較の定番
内積 (dot)向き＋長さ正規化済みならコサインと一致

正規化が効く

各ベクトルを長さ 1 に正規化（L2 正規化）しておくと、コサイン類似＝内積になり計算が単純化する。多くの埋め込みは正規化前提で扱う。

cos(a, b) = (a・b) / (|a|×|b|)   // |a|=|b|=1 なら a・b に一致

全件探索（brute force）

クエリと全候補のコサインを計算して上位 K を取る。実装は単純で厳密。N が数万程度なら現実的（とくに次元数が数百なら）。難点は N に比例する計算量。

graph LR Q[クエリ埋め込み] --> SC[全 N 件とコサイン] SC --> TOP[上位 K を選ぶ]

近似最近傍（ANN）

数百万〜規模では全件は重い。多少の取りこぼしを許す代わりに高速な索引を使う。

手法考え方
HNSW近いもの同士を結んだ多層グラフを辿る。高速・高精度で人気。
IVF空間をクラスタに分け、近いクラスタだけ調べる。
PQ（直積量子化）ベクトルを圧縮して距離計算を高速化（→ 第6章）。

メモリと I/O の現実

設計上の勘所 ベクトルは「1 件 × 次元 × バイト数」でメモリを食う。例: 512 次元 × 4 バイト（fp32）= 2KB/件 → 6.7 万件で約 134MB。端末では全件常駐が危険なので、ページング（少しずつ読む）や量子化（fp16/int8）で常駐を抑えるのが定石。

厳密 vs 近似の選び方

  • 件数が数万・次元が数百 → 全件探索で十分（実装も単純で取りこぼしなし）。
  • 数百万以上・低レイテンシ必須 → ANN。
  • メモリが厳しい → 量子化 + ページングを併用。
