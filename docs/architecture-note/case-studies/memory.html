事例: 写真枚数に比例したメモリ枯渇の解消と、その後の常駐圧縮

「写真ゼロの端末では起動するが、写真が多い端末では起動前に落ちる」。原因はメモリで、SwiftData の格納方法に潜んでいた。さらにその後、サムネ／フル画像キャッシュの常駐も圧縮した。学習材料として顛末を残す。

症状と切り分け

  • 写真ゼロの端末 → 起動する。
  • 写真が多い端末 → 起動できない。

「写真枚数に比例して何かが増える」と仮説。起動経路のうち、枚数に比例する処理を疑った。

真因

CLIP 埋め込み clipVector（512×Float32 ≈ 2KB/枚）を PhotoEnrichment の inline 属性として持っていた。SwiftData は fetch(FetchDescriptor<PhotoEnrichment>()) で行を丸ごと展開する。プロパティを読まなくても blob ごとロードされる。

致命的だった点 「軽量版」を謳う allEnrichedPhotosLite() も内部は全件 fetch → map。結果の構造体から vector を外しても、fetch の時点で 67k × 2KB ≈ 138MB を確保していた。生成・重複排除・prune など複数経路が同じ全件 fetch を行うため、起動直後にこれが重なり jetsam に至っていた。写真ゼロなら行ゼロ＝展開なし＝起動する、という観測と一致。

対策（採用したフルセット）

施策内容効果
別テーブル化clipVector を PhotoEmbedding 別 @Model へ分離。メタデータ fetch が blob に一切触れない。根本対処
Float16 化埋め込みを半精度で保存（2KB→1KB）。ClipMath.encodeHalf/decodeHalf。読み出し時に fp32 復元。DB・常駐を半減
ページング検索は enrichmentVectorPage(offset:limit:) でページ単位に読む。常駐をページ 1 枚分に
バッチ解放大量 upsert は writeChunk 件ごとに使い捨て ModelContext で save→解放。書き込み時の常駐を有界化
圧迫時停止メモリ圧迫中は背景タグ付けを一時停止（MemoryPressureMonitor）。jetsam 回避

学び

  • SwiftData の fetch は属性を丸ごと読む。大きな blob は別テーブルに逃がすか、読まない設計にする。
  • 「軽量版」と名付けても、fetch のコストは消えない。命名でなく実測（経路）で確かめる。
  • スキーマ変更は ModelConfiguration 名を採番して旧ストアを破棄（AutoAlbumV10）。埋め込みは背景で再生成され、精度は徐々に回復する。
  • 検証は実機スクショではなく、ビルド・ユニットテスト・診断ログのフットプリントで行う。

続編: 常駐の圧縮（埋め込み別テーブル化の“その後”）

構造的な枯渇（上記）を解消した後も、写真が多いと常駐が高めだった。精査すると複数の隠れ消費が見つかった。

原因内容対処
NSCache のコスト計上ズレ
（最大要因）		サムネのメモリ層にデコード済み画像を入れるのにコストを JPEG バイト（約20KB）で計上。totalCostLimit=100MB が JPEG 換算のため、実デコード（0.3〜2.4MB/枚）では約10倍以上＝〜1GB 相当まで保持し得た。insertDecoded/decodedCost（幅px×高px×4）で実コスト計上。Dropbox 側は totalCostLimit=48MB 併設。
サムネがフル解像度グリッドのセルを端末スケール（×3）で生成。ピンチの列数差で僅差サイズの重複デコードも増加。×2 上限＋64px bucket 量子化（1アセット1サイズ）。
フル画像がフル解像度ビューア（ズーム無し・scaledToFit）なのに 1 枚 40MB 超をデコード。ページャの前後保持でピーク増。ImageDownsampling（ImageIO・最大辺 1600）で表示時にダウンサンプル（保存は原バイト＝EXIF 保持）。
表示配列の常駐表示用 DropboxFileItem に contentHash（64桁hex）を 67k 件分常駐。表示には不要。cachedItems() で contentHash を渡さない（変更検知は別テーブル＋delta parser）。
上限が固定・圧迫対応が雑メモリ上限が固定値、メモリ警告時は全消去のみ。上限を端末 RAM 連動の Auto（約1.5%・40〜120MB）に。メモリ警告時は全消去でなく段階縮小→復帰で直近サムネを温存。
この続編の学び NSCache のコストは実バックストア（デコード後バイト）で計上する。 JPEG バイトで計上すると上限が桁で狂う。ズーム無しビューアはフル解像度をデコードしない（ImageIO ダウンサンプルでピーク削減）。長寿命の大規模配列に「表示に使わない文字列」を載せない。
