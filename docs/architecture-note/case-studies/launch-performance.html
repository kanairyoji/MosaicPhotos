事例: 起動の高速化とローディング表示

起動が遅い・真っ白な待ち時間がある問題に、「重い構築を描画パスから外す」「処理を時差で起動する」「1 秒超で読み込み表示」で対処した。

問題

各ストア（Dropbox / Backup / AutoAlbum）は init で ModelContainer を同期構築する。これを HomeView.init で行うと、最初の描画＝起動がブロックされる。さらに起動直後に複数の重い処理が同時に走り、スパイクを生んでいた。

対策 1: ストア構築を非同期化

RootView を新設し、起動直後に HomeStores.build() でストアを非同期構築する。各構築の合間に Task.yield() を挟んで主スレッドを解放し、その間にローディング表示のタイマーが動けるようにする。

sequenceDiagram participant Root as RootView participant Build as HomeStores.build() Root->>Root: スプラッシュ描画（即時） Root->>Build: 構築開始 loop 各ストア Build->>Build: ModelContainer 生成 Build-->>Root: Task.yield（主スレッド解放） end Note over Root: 1秒経過してもまだなら「Now loading…」 Build-->>Root: 完成 → HomeView 表示

対策 2: 1 秒超でローディング表示

別タスクで 1 秒タイマーを回し、それまでに準備できなければ ProgressView ＋「Now loading…」をフェードイン。高速起動ではローディングを出さない（チラつき防止）。スプラッシュ自体（アイコン＋アプリ名）は起動直後から出るので、真っ白な間がなくなる。

対策 3: 段階起動でスパイク回避

ホーム表示後のバックグラウンド処理を時差で開始する（HomeLifecycleTasks）。

タイミング処理
即時アルバムスキャン（キャッシュ即ロード）
+1.5s場所スキャン
+3s自動アルバム再生成・AI 再評価・背景タグ付け
+5sバックアップ記録ロード
学び 同期的な重い初期化は「最初のフレーム」を遅らせる。描画 → 非同期構築 → 差し替えの順にし、構築は yield で分割するとローディング UI が成立する。重い処理は「即時に必要か」で並べ替え、不要なものは遅らせる。
