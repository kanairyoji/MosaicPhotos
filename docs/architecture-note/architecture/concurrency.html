並行性とアクター分離

Swift Concurrency（async/await・actor）を全面採用。UI と状態は @MainActor、I/O や DB はアクター/バックグラウンドに逃がす。

MainActor 既定

アプリターゲットはビルド設定 SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor により、すべての型がデフォルトで @MainActor。UI 状態の更新は常にメインで行われ、データ競合を型レベルで防ぐ。

一方、PHImageManager や URLSession のコールバックはバックグラウンドスレッドで来る。これらは Task { @MainActor in ... } で明示的にメインへ戻す。

重い処理はメインから逃がす

写真ライブラリの全件列挙・座標集約・署名計算などは Task.detached(priority: .utility) でオフメイン実行し、結果（Sendable 値）だけをメインへ返す。

@ModelActor で DB を隔離

SwiftData の ModelContext はスレッド安全でない。AutoAlbumStore / DropboxCacheStore は @ModelActor（または actor）にして、コンテキストをアクタ専用 executor に束縛する。@Model はアクタ外へ漏らさず、必ず Sendable 値（EnrichedPhoto 等）に変換して返すのが鉄則。

@ModelActor
actor AutoAlbumStore {
    func allEnrichedPhotosLite() -> [EnrichedPhoto] {   // @Model ではなく値型を返す
        let records = (try? modelContext.fetch(...)) ?? []
        return records.map(\.asEnrichedPhoto)
    }
}

協調的な一時停止（背景処理を譲る）

背景の重い処理（CLIP 埋め込み・シーンタグ・顔スキャン）は、そもそもアプリを使っていない時間帯だけ動き（ADR-25。実行タイミング）、処理中もユーザー操作中・メモリ圧迫中・クラウド取得中は1 枚ごとに処理を譲る。共通の BackgroundTrickle 骨格が shouldPause を毎枚確認し、true の間は短いスリープで待機する。

設計判断 起動直後に「グリッド初期表示・Dropbox キャッシュ読み込み・AI 再評価」が同時に走るとメモリ/CPU スパイクが jetsam・watchdog を誘発する。そこで Task.sleep による段階起動（place scan を +1.5s、backup を +5s、重い AI を +3s）でピークをずらす。詳細は 起動の高速化。

テスト用 seam（DI）

外部依存はプロトコルで抽象化し、デフォルト引数で本番実装を注入する。HTTPClient（URLSession）/ DateProvider（時刻）/ AccessTokenProvider（トークン）はテストでスタブに差し替えられる。
