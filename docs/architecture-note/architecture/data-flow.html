データの流れと起動

起動からホーム表示、各バックグラウンド処理までの流れ。「即時に見せる」と「重い処理は遅らせる」を両立させる。

起動シーケンス

各ストアは init で ModelContainer を同期構築するため重い。これを最初の描画パスから外すため、RootView が起動直後に非同期で構築し、完成するまでスプラッシュを出す（1 秒を超えたら「Now loading…」）。

sequenceDiagram participant App as MosaicPhotosApp participant Root as RootView participant Stores as HomeStores.build() participant Home as HomeView App->>App: Diagnostics.install() App->>Root: 表示 Root->>Root: スプラッシュ描画（即時） Root->>Stores: 非同期構築（合間に Task.yield） Note over Root: 1秒超で「Now loading…」表示 Stores-->>Root: 各ストア完成 Root->>Home: HomeView(stores:)

ホーム表示後の段階起動

HomeView 表示後、バックグラウンド処理を時差で開始してピークを分散する（HomeLifecycleTasks）。

タイミング処理
即時アルバムスキャン（キャッシュ即ロード→無ければ走査）
+1.5s場所スキャン（ローカル + Dropbox 座標のグルーピング）
+3s自動アルバムの再生成・AI 再評価・背景タグ付けの開始
+5sバックアップ記録のロード（設定画面でのみ必要）

写真表示のデータフロー

ソース画面は共通の PhotoSourceContentView が状態（権限/通信/完了/失敗）で分岐し、グリッドと全画面を出す。サムネイル・フル画像は PhotoStore 経由で取得し、キャッシュ層を通す。

AI の付加情報フロー

写真は PhotoEnrichment（メタデータ）と PhotoEmbedding（CLIP 埋め込み・別テーブル）に分けて永続化される。検索は埋め込みをページングで読み、メモリを有界に保つ。

設計判断 「保存済みアルバムは即表示、重い再生成・埋め込みは遅延」という二段構えにより、体感起動を速く保ちつつ最新化も行う。埋め込みを別テーブル＋ページングにしたのはメモリ枯渇対策（事例）。
