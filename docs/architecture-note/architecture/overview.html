全体アーキテクチャ

「薄いアプリ層 + 役割ごとの SPM パッケージ」を基本方針に、ロジックと UI を分離する。アプリターゲットは組み立て（合成）だけを担い、機能は再利用可能なパッケージに収める。

3 つの設計原則

  1. ロジック層 / UI 層の分離（Core/UI 構成） — 端末写真と Dropbox は、SwiftUI 非依存の Core（LocalPhotoCore / DropboxCore）と、それに依存する UI（LocalPhotoKit / DropboxKit）に分ける。Core はテストしやすく、UI から独立して進化できる。
  2. 合成ルート（Composition Root） — 依存の結線はアプリ層の一点に集約する。AI の各 seam は AutoAlbumAdapters.swift で AutoAlbumEngine に注入する。
  3. インターフェイスの統一 — 写真ソースは PhotoSourceKit の PhotoStore / PhotoItem プロトコルに揃え、同じグリッド・全画面ビューを共有する。

なぜこの形か

設計判断 アプリ層を薄く保つことで、機能の大半がパッケージ単位でビルド・テストできる（純ロジックは macOS の swift test で高速に回る）。UI とロジックを分けるのは、PHImageManager や URLSession といった副作用を Core に閉じ込め、UI を差し替え可能にするため。設計判断の記録も参照。

レイヤをまたぐ規約

  • MainActor 既定 — アプリターゲットは SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor。詳細は並行性とアクター分離。
  • @Observable のみ — ObservableObject/Combine は使わない。
  • 設定キー・ロガーの一元化 — 各パッケージの専用 enum / LogChannel に集約。規約参照。
  • PhotoSourceKit は AutoAlbumCore に依存しない — フル画像の付加情報は photoInsight 環境クロージャ経由で受け取り、レイヤを逆流させない。
