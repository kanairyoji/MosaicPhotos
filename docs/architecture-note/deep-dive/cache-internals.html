キャッシュ実装詳細

プリミティブは ImageCacheKit に共通化し、破棄ポリシーは利用側が持つ。ここでは命名規則・無効化・LRU の実装まで踏み込む。

プリミティブ: DiskImageStore

FileManager を薄く包んだだけの I/O 層。破棄ポリシー（LRU）は持たず、LRU 列挙の材料だけ提供する。コアは Foundation のみ（macOS テスト可）、UIImage 便宜は #if canImport(UIKit)。

API役割
data(forName:) / write(_:name:) / remove(name:) / clear()基本 I/O
touch(name:date:)disk hit 時に mtime を更新（mtime ベース LRU 用）
entries()各ファイルの (name, size, mtime) を列挙
totalUsage()合計バイト数

2 つの利用側と別々のポリシー

graph TD ICK["ImageCacheKit
MemoryImageCache + DiskImageStore"] TC["LocalPhotoCore: ThumbnailCache (actor)
mtime ベース LRU"] DCS["DropboxCore: DropboxCacheStore (actor)
SwiftData(CacheUsageEntry) ベース LRU + 容量上限"] ICK --> TC ICK --> DCS
なぜポリシーを共通化しないか 端末写真は「再生成が容易・無効化条件なし」なので mtime LRU で十分。Dropbox は「contentHash で無効化」「容量上限の厳密管理」が要るため SwiftData で最終アクセスを持つ。要件が違うので「プリミティブ共通・ポリシー個別」に留める。

Dropbox キャッシュの構造

メタデータは SwiftData、バイナリは ImageCacheKit。保存先は Caches/DropboxKit/{thumbnails,fullimages}/。

ファイル命名（DropboxCacheNaming）

  • ファイル名 = SHA256(path) の 16 進 + 拡張子。
  • サムネイルは常に .jpg、本体は元拡張子（無ければ bin）。
  • 純ロジックなので actor から分離してテスト可能。

メタデータモデル

@Model主なフィールド役割
CachedDropboxItempath(unique) / contentHash / captureDate / latitude,longitude / cachedAtファイル一覧・座標・変更検知
CacheUsageEntrykind(thumbnail/fullImage) / バイト数 / 最終アクセスLRU・容量管理の台帳

無効化（contentHash）

Dropbox が返す contentHash がキャッシュ済みと異なれば、その path のサムネ/本体を無効化して次アクセスで再取得する。

graph LR REQ["path のサムネ要求"] --> CMP{contentHash 一致?} CMP -->|一致| HIT["キャッシュ採用
touch で最終アクセス更新"] CMP -->|不一致 / 無し| INV["旧バイナリ削除"] INV --> FETCH["API で取得 → 保存 → 台帳更新"]

LRU 容量管理

DropboxCacheStore+Eviction が CacheUsageEntry を最終アクセス昇順に並べ、種別ごとの上限（サムネ/本体）を超えた分を古い方から削除する。上限は設定（Storage 画面・DropboxCacheSettingsKeys）で変更可能。

メモリ層

MemoryImageCache は NSCache ラッパー。disk より前段で当たればディスク I/O を避ける。NSCache は OS のメモリ状況で自動破棄されるため、メモリ層自体の上限管理は OS に委ねる。
