AI アルバム実装詳細

自然文（任意言語）からアルバムを作る経路を、コードの登場人物と処理順で追う。設計の全体像はオンデバイス AIを先に。ここでは AIAlbumService を起点に「解釈 → 検索 → 審査 → 増分評価」の各フェーズを実装レベルで説明する。すべて AutoAlbumCore（SwiftUI 非依存・macOS でテスト可能）にある。

ライフサイクル：即時プレビュー → 夜間本番化

作成/編集（AIAlbumService.make）はLLM を一切呼ばない。決定的レイヤーだけの「プレビュー解釈」（AIAlbumInterpreter.previewInterpretation：レキシコン・日付パーサ・人物接地）で 1〜2 秒で結果を出し、pendingFinalization フラグ付きで保存する。FM 解釈と LLM 審査を含む本番化は夜間（finalizePending・電源＋Wi-Fi＋ロック中）に行う。UI を固めず、重い推論を夜間ゲートに寄せるための 2 段構えである（ADR-25/28）。

決定的レイヤーのみ・LLM なし"] PREV --> RS1["rankedSearch＋証拠ゲート"] RS1 --> SAVE1["保存: pendingFinalization=true
0 件でも保存する"] SAVE1 -.夜間 BGTask.-> FIN["finalizePending"] FIN --> INT["FM 解釈（interpretation）
QuerySpec＋英訳＋プローブ → 永続化"] INT --> RS2["rankedSearch（フル）"] RS2 --> GATE["証拠ゲート（除外つきのみ）"] GATE --> VER["LLM 審査（verified）"] VER --> REF{"0 件?"} REF -->|はい| PROBE["expandProbes → 1 回だけ再検索＋再審査"] REF -->|いいえ| DONE["GeneratedAlbum を upsert"] PROBE --> DONE
  • 0 件でも保存する：アルバムは“生きた条件”であり、索引が進めば夜間再評価で自然に埋まる（証拠主義）。
  • finalizePending はアルバムごとに BackgroundYield.heavyShouldPause() を確認し、ゲートが閉じたら（ロック解除等）残りを次回夜間へ持ち越す。
  • 再設定（検索文の変更）は interpreter.remove(id:) で解釈を破棄してやり直す。解釈のスキーマを変えたら SavedInterpretation.currentVersion を採番——版不一致の保存済み解釈は次回 interpretation() で自動再解釈される。

フェーズ 1：解釈（LLM は作成後の夜間に 1 回だけ）

interpretation(id:criteria:now:) は保存済みが現行版ならそれを返し、なければ FM で解釈して必ず決定的レイヤーで上書き接地する。小型 LLM の構造化出力は信用しない（ADR-23）：

  • 日付は RelativeDateParser（日英対応）が唯一の出典。LLM が出した日付範囲は捨てる。
  • 場所はライブラリの地名カタログ or 原文に実在するものだけ採用（幻覚地名の混入防止）。
  • 人物は PersonNameGrounder が命名済み顔クラスタへ接地。「太郎と花子」→ 山田太郎・山田花子（姓だけ・名だけの部分一致も解決）。
  • 視覚語は JapaneseVisualLexicon（決定的辞書）が頻出語と人物否定（「人が写っていない」等）を抽出し、LLM 出力とマージ。
  • QuerySpecSanitizer がプレースホルダ（"PLACE" 等）除去・カタログ丸写し検出・include/exclude 衝突解消を行う。
  • 言い換えプローブ（最大 4）もこのとき生成して永続化（ADR-35）。以降の再評価で LLM は走らない。

フェーズ 2：検索（searchWithPool）

本体は AIAlbumSearcher.searchWithPool。純ロジックなので macOS テストで固めてある（AIAlbumSearcherTests / MultiProbeTests 等）。処理順：

  1. ハード条件：QueryEvaluator.hardFilter。人物条件は peopleByRefKey（現在の顔クラスタ名）でライブ照合し、無ければ焼き込み値（EnrichedPhoto.people）へフォールバック。改名・統合が即反映される。
  2. 顔実測の除外：「人が写っていない」系の除外があるときは faceCounts（顔スキャン実測）で顔のある写真をハード除外。未スキャン分は CLIP 対比が受け持つ。
  3. タグのハード除外：除外語がシーンタグに離散一致する写真を落とす（閾値レス）。
  4. 肯定フレーズの選定：QueryEmbedder.phrase(include:exclude:semanticText:)。除外があるときは include 語だけを使う——全文には "without people" 等の否定が含まれ、CLIP は否定を理解せず逆に引っ張られるため。
  5. 意味採点：埋め込みは loadPage（PhotoEmbedding を refKey 昇順でページング）でストリームし、全件をメモリに持たない。採点は QueryEmbedder.semanticScore ＝「主フレーズ＋プローブの max-over-probes。除外語ベクトルのコサインが肯定側の最大値以上なら nil（相対判定・絶対閾値なし）」。除外つきのときはプローブを埋め込まない（肯定側の最大値が底上げされ除外が素通りした回帰の防止）。採用は上位から top − 0.06 の相対バンドのみ（フロア閾値は廃止）・最大 50 件。
  6. タグ一致ランキング：include 語とシーンタグの一致数降順（離散・閾値レス）。
  7. 字句ランキング：LexicalSearch.rank（地名・人物名のマッチ）。
  8. RRF 融合：HybridFusion.fuse([lexical, semantic, tagMatched], k: 60)。複数経路に現れる写真が上位に来る。
「空を返す」判断は実障害から決めた (a) フレーズ無し（翻訳保留等）でハード条件も無いときは空を返す——旧実装は無条件で base を返し「全 68,512 枚のアルバム」が生成された。(b) フレーズがあるのにタグ/意味/字句のどれにも当たらないときも空——旧実装は日付窓の全 7,508 枚へフォールバックし「子供」の意図が消えた。証拠主義（ADR-24）：索引が進めば自然に埋まる方を選ぶ。

フェーズ 3：証拠ゲートと LLM 審査

AIAlbumVerificationCoordinator が担う。

  • 証拠ゲート（evidenceGated・除外つきクエリのみ）：シーンタグ・顔実測・キャプションのいずれの証拠も無い候補を落とす。CLIP スコアだけの「もっともらしい誤検出」を審査に上げない。
  • オンデマンドキャプション（ADR-35）：審査対象のうちキャプション未生成の候補上位に、その場で VLM キャプションを生成する。予算は captionBudget = 40 枚/審査（500M VLM ≈ 4 秒/枚 × 40 ≈ 3 分弱・夜間実行）。生成分は台帳（TagsV1）へ保存され次回から再利用される。
  • 審査（verified）：候補ごとに証拠行（evidenceLine＝タグ・顔数・キャプション・日付/場所）を組み立て、AlbumVerifier（Foundation Models）が keep/drop/unsure を返す。unsure は最大 2 回再判定し majorityVerdict（純関数）で多数決。LLM が使えない環境では審査をスキップして融合結果をそのまま使う（劣化はするが空にはならない）。

フェーズ 4：再評価（増分＋ドリフト検知）

写真の取り込みが進んだときの更新は LLM を使わない。

  • 増分（refreshIncremental(newRefKeys:)）：新規に埋め込みが付いた写真だけを保存済みクエリベクトルで採点し、永続化されたスコアプール（refKey → コサイン・上位 300 件）へ mergePool でマージ → memberKeys(fromPool:) でメンバー再算出。全ページ再走査をしないので夜間バッチ内で安価に回る。採点規則は QueryEmbedder.semanticScore を共有し、フル検索と増分で結果が食い違わないことをテストで固定している。
  • ドリフト検知（refreshIfDrifted(threshold: 500)）：評価済み埋め込み数（evaluatedEmbedCount）と現在の埋め込み総数の差が閾値を超えたらフル再評価（refresh）を走らせ、増分の累積誤差をリセットする。

コンポーザーの入力支援は同じ決定的レイヤーの再利用

AutoAlbumEngine+Suggestions（ADR-37）。サジェストチップ（命名済み人物・カタログ地名・頻出タグ∩レキシコン・日付定型）は FaceStore / 地名カタログ / TagStore.topTags から、接地プレビューは previewInterpretation と同じ経路（PersonNameGrounder / JapaneseVisualLexicon.groundedPairs / RelativeDateParser）から作る。本番検索と同じ部品を使うことで「プレビューと実際の解釈がズレない」ことを構造的に保証する。ライブラリスナップショットは 5 分キャッシュ、プレビューは 350ms debounce。
