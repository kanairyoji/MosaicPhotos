並行・バックグラウンド処理の詳細

「UI を止めない」を最優先に、重い処理をアクタ/バックグラウンドへ逃がし、ユーザー操作とメモリ状態に応じて譲る。ここでは実際のパラメータと制御まで踏み込む。

アクタ分離の地図

区分実体役割
MainActor各ストア・エンジン（@Observable）UI 状態・公開 API
専用アクタAutoAlbumStore（@ModelActor）/ DropboxCacheStore（actor）/ PlaceNameResolver（actor）DB・キャッシュ・地名解決の直列化
detachedTask.detached(priority: .utility)全件列挙・グリッド集約・署名計算など CPU 仕事
backgroundTask(priority: .background)CLIP 埋め込みの trickle 実行

原則: @Model はアクタ外へ出さない。境界をまたぐ値は Sendable（EnrichedPhoto / AutoAlbumInfo / PlaceCandidate）に変換する。

背景 CLIP 埋め込みパイプライン

全写真の画像埋め込みは PhotoTagger.embedUnprocessed が担う。一度に全部処理せず、小バッチ＋バッチ間スリープで trickle する。

func embedUnprocessed(
    batchSize: Int = 8,
    betweenBatchNs: UInt64 = 2_500_000_000,   // 2.5s
    maxBatches: Int = 20_000,
    shouldPause: @MainActor () -> Bool = { false },
    onBatch: () async -> Void) async

未取得も sceneTagged=true] AP --> OB{16バッチ毎?} OB -->|はい| RA[onBatch → AI 再評価] OB -->|いいえ| SL RA --> SL[betweenBatchNs スリープ] SL --> CK

ポイント:

  • 未処理の抽出は limit 付き fetch（unembeddedRefKeys(limit:)）。全件ロードしない。
  • 取得不可でも sceneTagged=true にして無限ループを防ぐ（画像が無い/壊れた写真）。
  • onBatch は 16 バッチごとに呼び、AI アルバムを段階的に再評価（毎回やると重い）。
  • QoS は .background。UI 操作（.userInitiated）と CPU を奪い合わない。

「重さ」プリセット

バッチ件数と休止秒を 1 段階にまとめ、設定で選べる（BackgroundProcessing.presets）。

段階batchSizepauseSeconds用途
Very Gentle46.0最も端末に優しい
Gentle（既定）82.5通常
Balanced161.0速さ重視
Fast320.2充電中などで一気に

「譲る」条件（協調的一時停止）

shouldPause が true の間は 0.3s スリープで待機する。条件は 2 つ。

shouldPause: { isInteracting || MemoryPressureMonitor.shared.isUnderPressure }
  • isInteracting — グリッドのスクロール/スクラブ中。Coordinator の scrollViewWillBeginDragging 等で立つ。
  • isUnderPressure — メモリ圧迫中。Diagnostics のメモリ圧迫ソースが立て、約 20 秒後に自動解除。

Dropbox 側のスロットリング

  • サムネ並列数 — DropboxThumbnailBatcher がバッチで取得。並列数は設定（DropboxCacheSettingsKeys.thumbnailConcurrency）。
  • 初回同期の再代入間引き — ページごとに頻発する 6 万件規模の再フェッチ/再代入を scheduleCacheRefresh でスロットル（先頭即時＋以降 ~0.4s 間引き）。

段階起動（スパイク回避）

起動直後に重い処理が同時に走らないよう時差で開始する（HomeLifecycleTasks）。詳細は 起動の高速化。

設計の勘所 「速く終わらせる」より「UI と端末を圧迫しない」を優先。trickle・QoS・協調的一時停止・段階起動の 4 つで、重い AI 処理を体感ゼロに近づけている。
