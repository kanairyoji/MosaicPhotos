設計判断ハイライト（重要トップ 10）

本アプリで「効いた／学びになった」設計判断（ADR）から、特に重要な 10 個を抜粋して紹介する。各項は要点のみ。個別のバグ・性能対応の顛末（事例）は正本の case-studies.md と深掘りページへ。全件は Markdown のマスターを参照。

このページは“抜粋”、正本は Markdown 設計判断・事例の正本（マスター）はリポジトリ内の Markdown： records/decisions.md（ADR 全 37 件）と records/case-studies.md（事例 全件）。 本 HTML はそこから重要なものを取捨選択した派生物で、全件は転記しない。新しい判断・事例はまずマスターへ追記する。 詳しい顛末は 事例: メモリ / 事例: 起動 の深掘りページも参照。

設計 1. 検索の再設計＝「タグ台帳＋LLM 審査」（閾値レス） (ADR-24・ADR-23・ADR-1)

文脈: 当初は固定タグを捨て語彙ゼロの CLIP コサイン＋絶対閾値に一本化した（ADR-1）が、ライブラリ分布が不明で閾値が調整できず「除外が 97% を落とす／意味採用 0 件」が発生。小型 LLM の構造化出力も壊れた。
決定: (a) 解釈（検索文→QuerySpec・英訳）は作成時に 1 回だけ実行し永続化（ADR-23。旧・毎起動再解釈の 9.4s ハングを根治）。(b) 検索を Vision シーンタグ一致（離散・閾値レス）＋CLIP 対比＋字句の RRF 融合 → 証拠ゲート → LLM 審査（多数決）へ（ADR-24）。表示タグはシーンタグ＋ゼロショットで別途。
結果: 閾値依存がほぼ消滅し、判定根拠がタグ/キャプションで可視化。全写真へシーンタグ・CLIP 埋め込み・VLM キャプションを夜間に付与する索引が必要。

設計 2. ロジック層 / UI 層の分離（Core/UI） (ADR-3)

文脈: 副作用（PhotosKit/URLSession/SwiftData）を UI から切り離してテストしたい。
決定: *Core（SwiftUI 非依存）と *Kit（UI・@_exported import）に分割。
結果: 純ロジックを macOS で高速テスト。import は 1 つで済む。パッケージ数は増える（現在 11 個）。

設計 3. グリッドを UICollectionView に置換 (ADR-4)

文脈: 6 万件超で LazyVGrid + scrollTo のスクラバーが不安定。
決定: UIViewRepresentable で UICollectionView（diffable・プリフェッチ・contentOffset スクラバー）を採用。
結果: 大ジャンプが安定。UIKit のボイラープレートは増える。

設計 4. CLIP 埋め込みを別テーブル + Float16 (ADR-6)

文脈: 埋め込みを PhotoEnrichment に inline 格納し、全件 fetch のたび 138MB 級 blob を展開 → 写真の多い実機で起動クラッシュ。
決定: PhotoEmbedding 別テーブルへ分離し Float16 保存。メタデータ fetch は blob に触れない。
結果: 常駐が「ページ 1 枚分」に。スキーマ再構築（AutoAlbumV10）が必要。→ 事例。

設計 5. AI 検索を合成可能な QuerySpec（OR/NOT/多ファセット）へ (ADR-12)

文脈: 「ここ2年の子供」「京都か奈良の家族のお気に入り、スクショ除く」等、複雑条件・OR・NOT を柔軟に組みたい。
決定: DNF（節の OR・節内 AND・各条件 NOT）の QuerySpec。ハードは QueryEvaluator、相対日付は RelativeDateParser（日英）。
結果: 多ファセット＋OR を実現。QuerySpec は現在も解釈の内部表現として残り、採点は ADR-24 のタグ一致＋CLIP 対比＋LLM 審査へ発展した（旧フラット検索 API は撤去）。

設計 6. ModelContainer を自己修復で構築 (ADR-7)

文脈: SwiftData は破損・不整合で起動時 trap し、実機で原因不明クラッシュ。
決定: makeResilientContainer で「削除→再試行→インメモリ」とフォールバック。
結果: 起動を止めない。最悪データは失うが回復する（埋め込みは背景で再生成）。

設計 7. 重い処理は「使っていない時だけ」＋5 段階タイミング (ADR-25)

文脈: 旧ゲート「電源＋アイドル 60 秒」は、充電閲覧中に触らないと重い処理が走り出し、操作再開時にバッチ途中（最大約 27 秒）が譲れず固まった。
決定: アイドル判定を廃止し「電源＋低電力OFF＋アプリ非アクティブ＋Wi-Fi」へ。停止判定は全処理 1 枚単位。AI アルバム作成は即時プレビュー（決定的）→夜間本番化（LLM）の 2 段階。のちに実行タイミングを5 段階のユーザー設定（HeavyWorkTiming・純関数で判定）へ発展。
結果: フォアグラウンドで重い処理がゼロになり操作が軽い。初回全量索引は「充電＋Wi-Fi＋ロック」時間帯に限定され日数を要する（エスケープハッチ＝今すぐ処理）。→ バックグラウンド。

設計 8. 横断リファクタリング＝共通足場の抽出と責務分割 (ADR-26)

文脈: 機能追加で同じ骨格のコピーが層をまたいで蓄積（背景トリクル・自己修復コンテナ・Core ML ロード・BPE・UI 小物）。最大級の型に複数責務が同居。
決定: 挙動不変を絶対条件に共通足場（BackgroundTrickle/makeResilientModelContainer/CoreMLModelSupport/BPESupport 等）を抽出し、肥大型を分割（AIAlbumService・BackupEngine・DropboxThumbnailBatcher）。
結果: 譲りポリシー・コンテナ復旧・モデルロードの変更が各 1 箇所で済む。トークナイザは同梱語彙でビット等価を実測確認。

設計 9. モデル選定は「実測＋実機確認」で決める (ADR-31・ADR-32・ADR-34)

文脈: 画像認識が重く、CLIP / VLM の軽量化・置き換え候補を比較したい。カタログスペックでは判断できない。
決定: 共通ベンチ（Imagenette 実写＋Core ML 実行）でメモリ・速度・精度を実測して選ぶ。CLIP は INT8 量子化を採用（289→145MB・精度不変）。VLM は Florence-2-base を一度採用したが、実機 ANE/GPU の fp16 で cross-attention が破綻（全写真で無関係な文）し撤回、SmolVLM（decoder-only＝ANE 安全）へ回帰。のち SmolVLM-500M＋視覚のみ INT8 に更新し、最重量のキャプション生成はお気に入りのみに限定。
結果: 学びが 2 つ。(a) 量子化耐性は部品で非対称——連続ベクトル出力（視覚エンコーダ）は INT8 に強く、LLM デコーダは argmax が反転して壊れる。(b) Core ML の正しさは Mac だけでは検証できない——Mac の全 compute unit で正常でも実機 fp16 で壊れた。実測記録は records/model-evaluations.md。

設計 10. 検索品質は回帰ハーネスで数値検証してから直す (ADR-35)

文脈: 「自然文アルバムをもっと良く」——だが何が弱いか感覚では特定できず、直した“つもり”の劣化も検知できない。
決定: 正解ラベル付き評価セット（Imagenette 200 枚＋28 クエリ）で実パイプラインを走らせる SearchQualityTests を常設。ベースライン計測で「言い換えの取りこぼし」が弱点と数値で特定し、対策（LLM 言い換えプローブの max-over-probes ＋審査時オンデマンドキャプション）を同ハーネスの A/B で検証してから採用。
結果: 言い換え再現率 +17〜18pt を数値で確認。以後の検索変更は同ハーネスの回帰チェックを通す。実運用で見つかった穴（人物名 0 件・除外×プローブの相互作用）も事例として正本に記録。
