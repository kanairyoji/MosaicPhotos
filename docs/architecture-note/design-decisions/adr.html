設計判断・事例ハイライト（重要トップ 10）

本アプリで「効いた／学びになった」設計判断（ADR）と事例（バグ・性能対応）から、特に重要な 10 個を抜粋して紹介する。各項は要点のみ。全件は Markdown のマスターを参照。

このページは“抜粋”、正本は Markdown 設計判断・事例の正本（マスター）はリポジトリ内の Markdown： records/decisions.md（ADR 全 14 件）と records/case-studies.md（事例 全件）。 本 HTML はそこから重要なものを取捨選択した派生物で、全件は転記しない。新しい判断・事例はまずマスターへ追記する。 詳しい顛末は 事例: メモリ / 事例: 起動 の深掘りページも参照。

設計 1. 検索を語彙ゼロのオープン語彙 CLIP に一本化 (ADR-1)

文脈: 固定タグ方式は語彙外に弱く辞書保守も負担。
決定: 固定語彙・OCR を捨て、CLIP のオープン語彙＋構造化条件＋字句の RRF 融合へ。
結果: 自由なクエリに強い。代わりに全写真の画像埋め込み（背景処理）が必要。表示タグだけ別途ゼロショットで出す。

設計 2. ロジック層 / UI 層の分離（Core/UI） (ADR-3)

文脈: 副作用（PhotosKit/URLSession/SwiftData）を UI から切り離してテストしたい。
決定: *Core（SwiftUI 非依存）と *Kit（UI・@_exported import）に分割。
結果: 純ロジックを macOS で高速テスト。import は 1 つで済む。パッケージ数は増える（現在 11 個）。

設計 3. グリッドを UICollectionView に置換 (ADR-4)

文脈: 6 万件超で LazyVGrid + scrollTo のスクラバーが不安定。
決定: UIViewRepresentable で UICollectionView（diffable・プリフェッチ・contentOffset スクラバー）を採用。
結果: 大ジャンプが安定。UIKit のボイラープレートは増える。

設計 4. CLIP 埋め込みを別テーブル + Float16 (ADR-6)

文脈: 埋め込みを PhotoEnrichment に inline 格納し、全件 fetch のたび 138MB 級 blob を展開 → 写真の多い実機で起動クラッシュ。
決定: PhotoEmbedding 別テーブルへ分離し Float16 保存。メタデータ fetch は blob に触れない。
結果: 常駐が「ページ 1 枚分」に。スキーマ再構築（AutoAlbumV10）が必要。→ 事例。

事例 5. 写真枚数に比例したメモリ枯渇の解消

症状: 写真ゼロの端末は起動、写真が多い端末は起動前に落ちる。
原因: 「軽量版」allEnrichedPhotosLite() も内部は全件 fetch で、fetch 時点に 67k×2KB≈138MB を確保（複数経路が重なり jetsam）。
学び: SwiftData の fetch は属性を丸ごと読む。命名でなく実測（経路）で確かめる。→ 深掘り。

事例 6. メモリ常駐の圧縮（NSCache のコスト計上ズレ）

症状: 写真が多いと常駐が高く圧迫しやすい（ADR-6 後も残存）。
原因: サムネのメモリ層にデコード済み画像を入れるのにコストを JPEG バイトで計上 → 100MB 上限が実質約10倍（〜1GB 相当）に膨張。
対処: 実デコードサイズでコスト計上、サムネ ×2＋bucket、フル画像を ImageIO でダウンサンプル、メモリ警告で段階縮小、上限を端末 RAM 連動の Auto に。→ 深掘り。

設計 7. AI 検索を合成可能な QuerySpec（OR/NOT/多ファセット）へ (ADR-12)

文脈: 「ここ2年の子供」「京都か奈良の家族のお気に入り、スクショ除く」等、複雑条件・OR・NOT を柔軟に組みたい。
決定: DNF（節の OR・節内 AND・各条件 NOT）の QuerySpec。ハードは QueryEvaluator、内容語は CLIP でソフト採点。相対日付は RelativeDateParser（日英）。
結果: 多ファセット＋OR を実現。全滅回帰には「ハードで空なら内容のみへ緩和」の安全網で対処。

設計 8. CLIP 画像エンコーダを fp16（実機 ANE 優先） (ADR-11)

文脈: 画像埋め込みが 1 枚 0.5〜1.3 秒と遅い。原因は fp32 変換で Neural Engine に載らず GPU/CPU フォールバック。
決定: 変換を FLOAT16 に。fp16 はシミュレータで NaN 化し得るが有限性チェックで安全に無効化（画像タワーは実機検証）。
結果: 実機の埋め込みが ANE で高速化。シミュレータ最適化に引きずられない。

事例 9. All サムネが「ポツポツ」1 枚ずつ遅く出る

症状: All ビューで Dropbox サムネが 1 枚ずつポツポツ現れ非常に遅い。
原因: prefetch 未上書きで既定の直列 awaitを使用 → バッチャの集約（25枚/req）・並行（最大8本）が殺され実質バッチ 1・直列に。
対処: 先読みを並行発火してバッチャに集約させる。学び＝バッチ系ローダに「1 件ずつ await」は集約を無効化する。

設計 10. ModelContainer を自己修復で構築 (ADR-7)

文脈: SwiftData は破損・不整合で起動時 trap し、実機で原因不明クラッシュ。
決定: makeResilientContainer で「削除→再試行→インメモリ」とフォールバック。
結果: 起動を止めない。最悪データは失うが回復する（埋め込みは背景で再生成）。
